# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | Heroicons via Phoenix core components |
| Font | PT Serif for display/body emphasis, PT Sans for labels, HUD metadata, and controls |

### Existing Project Alignment

- Reuse the warm fantasy palette already defined in `assets/css/app.css`.
- Reuse the existing full-screen `Layouts.game/1` shell for in-world views.
- Keep interaction server-driven with focused JS hooks, matching the existing map hook pattern.

---

## Primary Screen Contract

### First-time entry screen

- Focal point: the `Enter World` button and its adjacent character/account preview card.
- Visual hierarchy: realm label first, character/account preview second, primary CTA third, settings cluster last.
- Mood: sparse MMO login screen, atmospheric rather than busy.
- Layout: vertically centered stack on mobile, with background atmosphere and a single framed entry panel.
- Settings cluster: compact secondary controls for sound, language, and help/account context only.

### Returning quick resume screen

- Focal point: current character summary and `Resume Journey` action.
- Visual hierarchy: character name and realm first, active state summary second, resume CTA third, secondary route to switch/recover last.
- The screen must feel lighter and faster than first-time entry.

### Notification deep-link entry

- Skip the resume screen entirely when the linked destination is still valid.
- If the destination is invalid, enter the closest valid shell context and show one brief inline notice explaining the redirect.

### In-game shell

- Focal point: the map viewport.
- Secondary focus: compact top HUD with character/context snapshot.
- Tertiary focus: bottom action bar for shell navigation.
- The map must visually dominate the screen; supporting chrome stays compact and low-noise.

### Recovery screens

- Focal point: problem heading plus one primary repair action.
- Visual hierarchy: what failed first, how to recover second, fallback to bot third.
- Recovery screens stay in-world visually and must never look like a generic browser error page.

---

## Component Inventory

- Atmospheric entry panel with realm label, crest/brand mark, character preview, and one primary CTA.
- Compact settings cluster with labeled icon buttons or short text buttons; no unlabeled icon-only actions.
- Quick resume card with character state summary and pending timer summary.
- Full-screen map surface using the existing Leaflet hook.
- Persistent top HUD showing character, realm/location, and one or two live status indicators.
- Bottom action bar with 3 to 5 labeled destinations.
- Inline status notice for deep-link redirects or successful auto-repair.
- Recovery card with primary retry and secondary open-bot fallback.

---

## Interaction Contract

- First open presents the entry screen before provisioning completes.
- Normal return presents the quick resume screen unless the player is already in a resolved shell context.
- Notification launches attempt direct restore into the target context before any shell gate.
- Pan and zoom are enabled on the map; marker inspection is allowed; travel actions are not present in Phase 1.
- Primary actions use full-width buttons on mobile.
- Icon-only controls require adjacent text labels or visible tooltip/caption fallback.
- Motion should be subtle: soft fade/slide on entry cards, gentle CTA glow, restrained HUD transitions.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 700 | 1.3 |
| Heading | 20px | 700 | 1.2 |
| Display | 32px | 700 | 1.05 |

### Typography Rules

- Use PT Serif for display text, headings, and atmospheric body copy.
- Use PT Sans for labels, metadata, timers, nav labels, and dense HUD text.
- Do not introduce more than these four font sizes in Phase 1.
- Do not introduce more than these two weights in Phase 1.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #0c0a09 | App background, map frame surround, full-screen atmospheric surfaces |
| Secondary (30%) | #1c1917 | Cards, HUD surfaces, bottom bar, overlays |
| Accent (10%) | #f59e0b | Primary CTA, active nav item, focus ring, key status emphasis |
| Destructive | #ef4444 | Session-reset warnings and unrecoverable failure actions only |

Accent reserved for: `Enter World`, `Resume Journey`, the active bottom-nav destination, keyboard/focus treatment, and the single highest-priority status emphasis on a screen

### Color Rules

- Maintain a 60/30/10 hierarchy across major surfaces.
- Do not use the accent color for every button or every marker.
- Default secondary actions use stone neutrals, not accent fill.
- Success/recovery confirmation may use the existing safe green as supporting feedback, but it is not a second accent.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Enter World |
| Empty state heading | Your path is quiet for now |
| Empty state body | No urgent world activity is waiting. Enter the shell to check your character, map, and next destination. |
| Error state | We couldn't restore your traveler seal. Try again from Telegram, or reopen MMGO from the bot to refresh your link. |
| Destructive confirmation | Reset linked session: This will drop the current app session and ask Telegram to identify you again. |

### Additional Copy Rules

- Resume CTA label: `Resume Journey`.
- Recovery primary action label: `Try Telegram Again`.
- Recovery secondary action label: `Open Bot Fallback`.
- Invalid deep-link notice: `That destination has already changed. We brought you to the nearest valid view.`
- Settings labels should stay plain and short: `Sound`, `Language`, `Help`.
